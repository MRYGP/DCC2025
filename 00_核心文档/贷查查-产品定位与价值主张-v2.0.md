## 一、一句话定位

### 1.1 对外定位（公开）

> **"贷查查 = 你的贷款能力Dashboard"**  
> 银行政策在变，你的条件在变，你的贷款能力也在变。随时测一测，不错过任何机会。

### 1.2 内部定位（核心）

> **"贷查查是我们业务体系的前端入口 + 合伙人的专属武器"**  
> 不是开放的行业基础设施，而是有控制的漏斗系统。

## 二、贷查查的双重身份

### 2.1 对外：免费的贷款体检入口

**功能**：
- 企业主可以免费测
- 传播快、用户多
- 但只给"概要"，不给"详情"

**目的**：

- 让客户主动联系我们

### 2.2 对内：合伙人的专属武器

**功能**：

- 合伙人用来获客、说服客户
- 客户进入我们的体系

**目的**：
- 让合伙人有独特价值
- 客户流向我们的后端服务
- 形成商业闭环

## 三、真实差异化：银行给不了的东西

### 3.1 四层价值体系

| 层次 | 价值 | 银行能给吗 | 我们能给吗 |
|------|------|-----------|-----------|
| **第1层** | 单银行额度预测 | ✅ 能 | ✅ 能 |
| **第2层** | 跨银行额度对比 | ❌ 不能 | ✅ 能 |
| **第3层** | 组合策略设计 | ❌ 不能 | ✅ 能 |
| **第4层** | 改造前后对比 | ❌ 不能 | ✅ 能 |

**银行只能做第1层，我们能做到第4层。这才是真正的差异化。**

### 3.2 组合策略示例

```
企业情况：年纳税8万，年开票300万，有房产估值400万

银行A只会说："你在我行能贷税贷80万"
银行B只会说："你在我行能贷发票贷60万"
银行C只会说："你在我行能贷抵押贷280万"

贷查查能给出的整体方案：

┌─────────────────────────────────────────────────────────────────┐
│  方案A（保守-低息优先）：                                         │
│  - 建行云税贷：80万（利率3.15%）                                 │
│  - 总额：80万，综合利率3.15%                                     │
│  - 适合：不急用钱，追求低成本                                    │
├─────────────────────────────────────────────────────────────────┤
│  方案B（平衡型）：                                                │
│  - 建行云税贷：80万（利率3.15%）                                 │
│  - 交行发票贷：50万（利率4.5%）                                  │
│  - 总额：130万，综合利率3.68%                                    │
│  - 适合：需要一定额度，又想控制成本                               │
├─────────────────────────────────────────────────────────────────┤
│  方案C（激进-额度优先）：                                         │
│  - 建行云税贷：80万（利率3.15%）                                 │
│  - 交行发票贷：50万（利率4.5%）                                  │
│  - 招行抵押贷：200万（利率2.98%）                                │
│  - 总额：330万，综合利率3.21%                                    │
│  - 适合：急需大额资金，有抵押物                                  │
└─────────────────────────────────────────────────────────────────┘
```

### 3.3 改造前后对比示例

```
当前状态：年开票100万，年纳税2万
当前可贷：约30万（微众微业贷）

┌─────────────────────────────────────────────────────────────────┐
│  如果做开票提升（H业务）：                                        │
│  - 改造后：年开票500万，年纳税10万                               │
│  - 改造周期：6个月                                               │

│  - 改造后可贷：约150万（税贷80万 + 发票贷70万）                   │
│  - 增加额度：120万                                               │
│  - ROI：增贷120万 vs 成本X万                                     │
├─────────────────────────────────────────────────────────────────┤
│  如果做高新认定：                                                 │
│  - 改造后：获得高新技术企业资质                                   │
│  - 改造周期：8-12个月                                            │
│  - 改造成本：约X万                                               │
│  - 改造后可贷：约200万（高新企业贷）                              │
│  - 增加额度：170万                                               │
│  - ROI：增贷170万 vs 成本X万                                     │
└─────────────────────────────────────────────────────────────────┘
```

### 3.4 为什么银行不能提供这些？

| 内容 | 银行为什么不能提供 |
|------|-------------------|
| 跨银行对比 | 银行只推自己的产品，不会告诉你去别家 |
| 组合策略 | 银行不会帮你设计"在我这贷一点，在别家贷一点" |
| 改造建议 | 相当于教客户"怎么包装"，监管红线 |
| 改造前后对比 | 暗示客户"做XX就能多贷"，银行不能说 |

## 四、分层开放设计

### 4.1 公开版（企业主/任何人）

**能看到**：
- 基础诊断结果
- 体质分级（A/B/C）
- 当前可贷额度预测
- 跨银行产品对比
- 组合策略建议
- 改造建议（概要）："如果提升XX，预计可贷XX万"

**看不到**：
- 改造方案详情（具体怎么做、成本、周期）
- 后端服务商对接

**关键入口**：
- 每份报告都有"想了解详情？联系我们的顾问"

### 4.2 合伙人版（专属）

**全部功能**：
- 公开版所有内容
- 改造方案完整详情（步骤、成本、周期、ROI）
- 直接对接我们的后端服务商
- 客户管理功能
- 历史记录对比

### 4.3 为什么这样设计？

| 设计 | 目的 |
|------|------|
| 公开版不给详情 | 想知道"怎么改造"必须联系我们 |
| 报告里有入口 | 即使别人用了，客户也可能来找我们 |
| 合伙人专属版 | 让合伙人有独特价值，客户进我们体系 |

## 六、商业闭环

### 6.1 完整流程图

```
企业主自己测（公开版）          合伙人给客户测（专属版）
      │                              │
      ↓                              ↓
   贷查查诊断系统（基础诊断+改造概要）
      │                              │
      │想了解详情                     │有完整方案
      ↓                              ↓
  【联系我们的顾问】←────────────────┘
      │
      ↓
   我们的客服/顾问
   - 企业主自己来的 → 分配给合伙人
   - 合伙人带来的 → 直接对接后端
      │
      ↓
   我们的后端服务商
   - 资质认定（科小/高新）
   - 带量服务（H业务）
   - 银行渠道对接
      │
      ↓
   成交 → 分润
```

### 6.2 关键控制点

| 控制点 | 设计 | 目的 |
|--------|------|------|
| 公开版不给详情 | 只说"可以"，不说"怎么" | 必须联系我们 |
| 报告里有入口 | 每份报告都有 | 客户可能自己来 |
| 后端绑定 | 只对接我们的服务商 | 不流失到外部 |

## 七、商业模式与盈利来源

### 7.1 两种模式的本质区别

| 模式 | 我们的角色 | 赚什么钱 | 投入 | 风险 |
|------|-----------|---------|------|------|
| **模式A：广告平台** | 流量入口 | 广告费（固定） | 低 | 低，但收入有限 |
| **模式B：业务中台** | 客户筛选+分发 | 返佣（按成交） | 高 | 高，但收入天花板高 |

### 7.2 当前阶段策略：模式B为主

**为什么现阶段不做广告？**

| 原因 | 说明 |
|------|------|
| 流量不够 | 没流量，广告卖不上价 |
| 口碑未建立 | 后端不信任，不愿意付广告费 |
| 影响体验 | 广告太多，用户觉得不专业 |
| 短视 | 广告费是小钱，返佣才是大钱 |

**策略**：
- **0-6个月（现在）**：模式B为主，不做广告
  - 先跑通闭环，积累案例
  - 建立口碑，验证商业模式
- **6-12个月**：模式B为主 + 少量广告
  - 有流量后可以卖广告位
- **12个月后**：混合模式
  - 广告做基础收入，返佣做增长

### 7.3 收入来源矩阵

| 收入来源 | 付费方 | 收费方式 | 优先级 |
|---------|--------|---------|--------|
| 银行贷款返佣 | 银行/渠道 | 贷款额0.5-1% | P0 |
| H业务分润 | H业务方 | 月息分润20-30% | P0 |
| 资质认定返佣 | 认定机构 | 服务费10-20% | P0 |
| 流水服务分润 | 服务方 | 月息分润20-30% | P1 |
| **贷款置换服务费** ⭐新 | 企业主 | 按节省利息分成 | P1 |
| **担保增信返佣** ⭐新 | 担保公司 | 按担保额/费 | P1 |
| **综合融资方案设计费** ⭐新 | 企业主 | 按方案复杂度 | P1 |
| 广告费 | 后端服务商 | 月费/年费 | P2（以后再做） |
| 合伙人会费 | 合伙人 | 年费/培训费 | P2（以后再做） |

### 7.4 长期业务 vs 一次性业务

**一次性业务**（高新/科小/专精特新）：
- 客户做一次就完了
- 返佣一次性到账
- 需要持续获客
- 收入波动大

**长期业务**（H业务/流水服务）：
- 客户持续使用（6-12个月）
- 每月都有返佣
- 一个客户 = 6-12个月收入
- 收入稳定，可预测
- ⚠️ 但风险也高（客户跑路、后端出问题）

**长期业务的复利效应**：
- 客户累积，收入持续增长
- 第1月：2个客户，月收入1万
- 第6月：12个客户，月收入6万
- 第12月：20+个客户，月收入10万+

### 7.5 专人的必要性

**为什么需要"专人"？**

| 工作内容 | 说明 |
|---------|------|
| 筛选客户 | 不是所有客户都适合，要先判断 |
| 匹配后端 | 不同客户适合不同服务商 |
| 跟进成交 | 客户犹豫时要推一把 |
| 售后服务 | 出问题时要协调 |
| 风险控制 | 防止后端坑客户、客户坑后端 |

**没有专人 = 纯粹做流量 = 只能收广告费**  
**有专人 = 业务中台 = 能收返佣**

### 7.6 后端管理原则

| 原则 | 说明 |
|------|------|
| 宁缺毋滥 | 每个品类只合作1-2家靠谱的 |
| 先小后大 | 先带几个小客户测试，再大量合作 |
| 合同先行 | 返佣比例、结算周期、违约责任写清楚 |
| 定期对账 | 每月对账，有问题及时沟通 |
| 客户反馈 | 收集客户对后端的评价，淘汰差的 |

### 7.7 主要风险和应对

| 风险 | 说明 | 应对 |
|------|------|------|
| 后端跑路 | 后端收了钱不服务/服务差 | 选靠谱的后端，分批付款 |
| 后端绕过我们 | 后端直接联系客户，不给返佣 | 合同约束，建立信任 |
| 客户投诉 | 客户对服务不满意，找我们 | 把好筛选关，选好后端 |
| 返佣拖欠 | 后端拖着不给返佣 | 合同约定，按月结算 |
| 长期业务客户跑路 | H业务客户中途不做了 | 风险告知，不承诺收益 |

### 7.8 盈利模式总结

**一句话**：
> 贷查查通过"诊断引流 + 专人筛选 + 后端分润"赚钱  
> 不是卖广告位的流量生意，而是业务中台的服务生意

**收入结构目标**：

| 收入来源 | 占比 | 特点 |
|---------|------|------|
| 银行贷款返佣 | 30% | 一次性，客单价高 |
| H业务/流水分润 | 40% | 长期，稳定，可累积 |
| 资质认定返佣 | 20% | 一次性，利润高 |
| 银行渠道返佣 ⭐新 | 纳入银行贷款返佣 | 精准客户，完成KPI |
| 广告费（以后） | 10% | 稳定，但天花板低 |

**关键成功要素**：
- 流量：贷查查要有足够的体检用户
- 专人：要有懂业务的人做筛选和对接
- 后端：要有靠谱的后端服务商
- 系统：要有客户管理系统，跟踪长期业务

### 7.9 受益者矩阵

**受益者全景图（5类角色）**：

| 受益者 | 受益内容 | 出钱方式 | 出力方式 | 优先级 |
|--------|---------|---------|---------|--------|
| **银行** ⭐新 | 精准客户、完成指标 | CPS返佣、合作费 | 数据反馈、政策更新 | P0 |
| **后端服务商** | 获得客户 | 返佣、广告费 | 服务交付 | P0 |
| **外部中介** | 专业工具 | 贴牌费 | 数据众包 | P1（未来） |
| **合伙人** | 工具+后端 | 早期免费 | 获客、服务 | P0 |
| **企业主** | 信息+服务 | 早期免费 | 留资、转介绍 | P0 |

### 7.10 银行作为新角色

**银行的痛点**：
- 小微贷款是政治任务必须做
- 但获客难、审批累、坏账高
- 需要精准客户，提高通过率

**贷查查的价值**：
- 帮银行前置筛选，只推符合条件的客户
- 提高通过率，降低审批成本
- 完成小微贷款KPI指标

**合作模式**：
- **渠道返佣**：推客户给银行，按成交额收返佣
- **数据合作**：银行反馈审批结果，我们校准模型
- **深度合作**：API对接，直接推荐客户

**切入路径**（分阶段）：
1. **渠道阶段**：先做渠道，推客户收返佣
2. **数据阶段**：有量后换数据，审批结果校准模型
3. **深度合作阶段**：规模后API对接，直接推荐

**"出力换优惠"机制**：
- 银行反馈审批结果 → 换优先推荐位
- 银行提供政策更新 → 换更多客户推荐
- 数据贡献越多，合作越深入

### 7.11 服务分层设计

**三层服务体系**：

| 层级 | 服务内容 | 交付方式 | 客户类型 | 收入模式 |
|------|---------|---------|---------|---------|
| **Layer 1：标准服务** | 诊断报告、常规产品推荐 | 线上，公开 | 所有用户 | 免费引流 |
| **Layer 2：增值服务** | 体质改造、组合融资、贷款置换 | 线下，半公开 | 有需求的用户 | 服务费/分润 |
| **Layer 3：深度服务** | 特殊项目对接、资源整合、定制方案 | 纯线下，不公开 | 有资源的用户 | 方案设计费/分润 |

**Layer 1：标准服务（线上，公开）**
- 诊断报告生成
- 常规产品推荐
- 基础额度预测
- 改造建议概要

**Layer 2：增值服务（线下，半公开）**
- 体质改造方案（H业务、资质认定）
- 组合融资设计
- 贷款置换服务
- 担保增信对接

**Layer 3：深度服务（纯线下，不公开）**
- 特殊项目对接
- 资源整合撮合
- 定制化方案设计
- 需要面对面深度沟通

**转化漏斗**：
```
所有体检用户（100%）
    ↓ 10%对"特殊路径"感兴趣
主动联系咨询（10%）
    ↓ 30%经沟通发现有资源/有意向
进入深度服务（3%）
    ↓ 50%最终成交
成交（1.5%）
```

### 7.12 启发式方案设计

**核心逻辑**：
> 不是"我们能帮你做什么"  
> 而是"如果你有XX，我们可以帮你实现YY"

**什么是启发式方案？**

| 类型 | 特点 | 示例 |
|------|------|------|
| **确定性方案** | 基于现有条件，给出确定结果 | "你符合XX条件，可以申请XX贷款，额度约XX万" |
| **启发式方案** | 展示可能性，探测客户资源 | "如果您有XX资源/能力，可能可以实现YY" |

**启发式方案的价值**：

| 价值 | 说明 |
|------|------|
| **筛选客户** | 有资源的客户会主动说"我有这个" |
| **启发思路** | 客户可能没想到自己有这种可能性 |
| **降低承诺风险** | 我们没有承诺，只是展示可能性 |
| **为深度合作埋伏笔** | 客户说"我有"，我们才深入 |
| **展示专业度** | 知道这些"可能性"本身就是专业 |

**话术边界**：

| 场景 | ✅ 可以说 | ❌ 不能说 |
|------|---------|---------|
| 展示可能性 | "如果您有XX，可能可以..." | "我们保证能帮你拿到..." |
| 探测资源 | "您这边有没有XX方面的资源？" | "我们可以帮你搞定XX" |
| 解释路径 | "有些企业通过XX方式实现了YY" | "我们和银行有内部关系" |
| 约线下 | "这个需要具体了解您的情况" | "来了就给你办" |

**体检报告的"特殊路径提示"模块**：


## 八、未来模式：贴牌SaaS（Phase 2）

> **Phase 2（6-12个月后）**：产品成熟后，开始贴牌扩张

### 8.1 核心洞察

**贴牌模式的精髓不是"卖软件"，而是"软件换数据"**：

- 我们提供：诊断系统、产品库框架、算法逻辑、持续更新维护
- 他们提供：当地银行政策数据、实时政策变化、地域差异信息、实测反馈

**类比**：高德地图的众包模式
- 用户贡献路况数据 → 高德地图更准
- 中介贡献银行数据 → 贷查查更准

### 8.2 两条路线对比

| 维度 | 自营闭环（Phase 1） | 贴牌SaaS（Phase 2） |
|------|-------------------|-------------------|
| 我们的角色 | 业务中台 | 软件供应商 |
| 收入来源 | 返佣（按成交） | 软件费（按月/年） |
| 收入特点 | 波动大，但单笔高 | 稳定，但单笔低 |
| 客户归属 | 我们的 | 中介的 |
| 投入 | 重（需要专人跟进） | 轻（只做软件） |
| 规模化 | 慢（每个客户都要服务） | 快（软件可复制） |
| 数据来源 | 自己调研 | 中介众包 ✨ |

### 8.3 两条路可以共存

**核心区域（湖南）**：自营闭环
- 发展我们的合伙人
- 收返佣，深度运营
- 控制客户流向

**外围区域（全国）**：贴牌SaaS
- 向外地中介公司提供白标版本
- 收软件费，换数据
- 轻资产扩张

```
                    ┌─────────────────────────┐
                    │      贷查查中央系统      │
                    │   • 核心算法            │
                    │   • 基础产品库          │
                    │   • 数据汇总            │
                    └───────────┬─────────────┘
                                │
            ┌───────────────────┼───────────────────┐
            ↓                   ↓                   ↓
    ┌───────────────┐   ┌───────────────┐   ┌───────────────┐
    │  长沙（自营）   │   │  成都（贴牌）  │   │  杭州（贴牌）  │
    │  我们的合伙人   │   │  当地中介公司  │   │  当地中介公司  │
    │  收返佣        │   │  收软件费      │   │  收软件费      │
    │  深度运营      │   │  贡献当地数据  │   │  贡献当地数据  │
    └───────────────┘   └───────────────┘   └───────────────┘
```

### 8.4 数据众包的战略意义

**解决了"实时更新"的难题**：

- 之前：我们说要"实时更新银行政策"，但没说怎么做到
- 现在：中介天天跑银行，他们最知道政策变化
- 方案：通过数据贡献激励机制，让中介主动更新

**数据越全，产品越准，所有用户都受益**。

### 8.5 分阶段策略

| 阶段 | 策略 | 原因 |
|------|------|------|
| **Phase 1（0-6月）** | 先跑通自营模式 | 还没有验证产品，卖不出去 |
| **Phase 2（6-12月）** | 开始尝试贴牌 | 产品成熟了，可以复制 |
| **Phase 3（12月后）** | 大规模推贴牌 | 有案例、有口碑，好卖 |

### 8.6 需要解决的问题

| 问题 | 解决方案 |
|------|---------|
| **区域冲突** | 核心区域（湖南）不卖贴牌，只发展自己的合伙人；或核心区域的贴牌费更高，且要求不做H业务等核心业务 |
| **数据质量** | 交叉验证（同一城市多个中介的数据互相验证）、实测追踪（记录预测vs实际）、积分体系（数据准确的中介获得更多积分） |
| **防止抄袭** | 核心算法不开放，只提供结果；持续更新，他们抄不完；数据壁垒（越多中介贡献数据，数据越全，单个中介抄不走） |

### 8.7 一句话总结

> **贴牌SaaS不是替代自营闭环，而是第二增长曲线：**
> - **自营闭环**：在核心区域深耕，收返佣，控制客户
> - **贴牌SaaS**：在外围区域扩张，收软件费，换数据
> 
> **两条腿走路，既有深度，也有广度。**

## 九、公信力来源

### 8.1 核心逻辑

> **公信力 = 预测准确率**  
> "贷查查说能贷120-150万，实际批了130万" → 这就是公信力

### 8.2 积累机制

- 每个实测案例都记录"预测vs实际"
- 100个案例后，可以对外说"预测准确率XX%"
- 这比任何概念包装都有说服力

### 8.3 验证清单

| 待验证项 | 验证方式 | 状态 |
|---------|---------|------|
| 单产品额度预测准确率 | 公司A实测 | ⏳ 待验证 |
| 组合策略可行性 | 多银行同时申请 | ⏳ 待验证 |
| 改造前后对比准确性 | 改造后再测 | ⏳ 待验证 |

## 十、诚实反思：哪些是真的，哪些还需验证

> **本章为内部视角，不对外展示。目的是避免"自我合理化"，保持清醒。**

### 8.1 残酷的问题

**客户为什么来找你？**

| 我们说的 | 可能的真相 |
|---------|-----------|
| "想做贷款体检" | ❌ 没人会这么想 |
| "想了解自己贷款能力" | ❓ 可能，但不是主要动机 |
| "想贷款，不知道找谁" | ✅ 这才是真实需求 |
| "被其他中介坑过" | ✅ 这才是痛点 |

**真相**：客户要的是**"能贷到钱"**，不是"体检报告"。

### 8.2 系统的真实价值

**问**：如果没有贷查查系统，你还能做这门生意吗？

**诚实答案**：能。靠经验、关系、行业理解也能做。

**那系统的价值是什么？**
- ✅ **放大器**：让一个人能服务更多客户
- ✅ **标准化**：让新人也能达到及格线
- ✅ **可复制**：让合伙人能用
- ❌ **不是发动机**：发动机是经验和判断

### 8.3 价值主张验证状态

| 我们说的 | 验证状态 | 说明 |
|---------|---------|------|
| "诊断更准" | ❌ 未验证 | 没跑过实测 |
| "产品更全" | ❌ 未验证 | 才20个产品 |
| "服务更透明" | ❓ 待验证 | 客户真的在意吗？ |
| "飞轮效应" | ❌ 未验证 | 第一圈都没转 |

### 8.4 三个版本的定位

**对外版（包装过的）**：
> "我们是贷款体检医生，帮企业做体质诊断。"

**内部版（诚实的）**：
> "我们在尝试把老中介的经验整理成系统，目前还在验证。"

**务实版（对客户说的）**：
> "我们帮你搞清楚能贷多少、在哪贷、怎么组合最划算。"

### 8.5 当前最重要的事

1. 用公司A跑完整流程，验证预测准确性
2. 积累"预测vs实际"案例
3. 验证组合策略可行性

## 十二、一句话总结

> **贷查查不是开放的行业基础设施，而是有控制的漏斗系统：**
> - **公开版**：免费引流，给概要不给详情
> - **合伙人版**：专属武器，完整功能+后端对接
> - **商业闭环**：企业主→贷查查→联系我们→后端→分润
> - **真实差异化**：跨银行对比+组合策略+改造前后对比（银行给不了）
> - **公信力**：来自预测准确率，不是概念包装
