## 一、文档定位与总体目标

### 1.1 文档定位

本文件是 **「资金体检 + 贷款体质医生」产品线的会计实习生工作执行计划**，用于：

- 让实习生清楚知道：每天、每周具体要做什么；  
- 让技术实习生知道：产品数据会怎么被整理，方便接入系统；  
- 让松阳随时能追踪进度、调整优先级。

### 1.2 总体目标

> 把现有分散在本地的银行产品资料（Word / PDF / Excel / 数据库1 / 中介产品表），  
> 在 1–3 个月内整理成一套 **结构化、可查询、可被贷查查调用的“银行产品知识库”**。

具体要达成 4 个结果：

1. 所有银行产品的原始资料 **有统一目录、有索引、可追溯来源**；  
2. 所有要用的产品，**都录入到一张统一的《产品总表.xlsx》** 中；  
3. 对重点产品（税贷/发票贷/流贷等），有至少一轮 **纸面测试记录**，部分有真实案例记录；  
4. 技术实习生可以基于《产品总表》构建程序侧产品库（JSON/数据库），支撑贷查查的产品匹配逻辑。

## 二、角色分工（谁负责什么）

### 2.1 松阳（产品 owner / 首席医生）

- 决定产品方向 & 哪些产品是「重点测试对象」；  
- 定义诊断逻辑和体质分级标准（A/B/C）；  
- 决定哪些指标必须录入产品库（额度规则、票税要求等）；  
- 参与与银行/中介的关键沟通，确认重要口径。

### 2.2 技术实习生（Agent / 系统工程师）

- 负责把《产品总表.xlsx》 → 程序可用的数据结构（JSON / DB）；  
- 在 `DCC2025` 仓库中实现：  
  - 产品查询接口  
  - 根据体检结果匹配产品的算法/规则引擎；  
- 和会计实习生一起定义字段含义（命名、类型、取值范围）。

### 2.3 资金体检产品实习生（会计专业）

一句话定位：

> **银行产品档案管理员 + 财税视角产品分析助手。**

核心职责：

1. 把所有银行产品的原始资料，整理进统一目录，建好「原始资料索引表」；  
2. 按统一字段，逐个产品录入到《产品总表.xlsx》；  
3. 主动联系银行客户经理/中介，补齐关键字段；  
4. 按标准企业画像，对重点产品做纸面测试；  
5. 记录真实案例数据，形成《产品实战案例》。

## 三、工作内容模块（WHAT）

> 下列 A–E 模块，是这个岗位的长期工作框架。

### 模块 A：产品资料收集 & 归档

**目标：**原始资料不丢、不乱，随时能找到上游文件。

1. 将现有散落在本地的文件，迁移到 `DCC2025/01_银行政策库/01_原始资料/` 目录下，按以下规则分类：  
   - 一级：银行名（如：`交通银行/`、`民生银行/`、`农业银行/`、`微众银行/` 等）；  
   - 二级：产品名 + 来源 + 日期（如：`交行-企业贷产品大纲-202412.xlsx`）。  
2. 同步整理现有 Excel：  
   - `00.产品汇总.xlsx`  
   - `线下产品.xlsx`  
   - `查询要求速览.xlsx`  
   - 其他单独银行的产品大纲（如 `交通-企业贷产品大纲.xlsx`、`民生银行-民生惠.xlsx`、`农行-纳税e贷产品大纲.xlsx` 等）。  
3. 维护一张《原始资料目录表》（Excel），列字段：  
   - bank_name（银行）  
   - product_name（产品名）  
   - source_type（中介 / 银行直 / 官网 / PPT / 其他）  
   - file_name（文件名）  
   - file_path（在仓库中的路径）  
   - last_update_date（文件时间）  
   - status（初步录入 / 已完整录入 / 废弃）  

### 模块 B：标准字段录入（产品总表）

**目标：**所有可用产品都以统一字段形式存在，便于查询和算法使用。

1. 以现有 `00.产品汇总.xlsx` 为基础，升级为统一的《产品总表.xlsx》，一行 = 一个产品。  
2. 字段示例（可根据松阳/技术实习生后续补充）：  
   - 基本信息：  
     - bank_name，product_name，product_type（税贷/发票贷/流贷/抵押贷/政策性/其他），online_or_offline，region；  
   - 客户主体：  
     - target_customer（小微/个体/一般纳税人/高新/专精特新等），min_operation_years，taxpayer_type（一般/小规模），禁入行业；  
   - 额度 & 利率：  
     - quota_formula（文字说明），quota_min/quota_max，rate_min/rate_max，term_min/term_max，repayment_mode；  
   - 关键财务指标：  
     - invoice_requirement（近12/24个月开票要求），tax_requirement（近12/24个月纳税要求），revenue_requirement，cashflow_requirement（对公流水等）；  
   - 抵押/担保/征信：  
     - credit_type（信用/抵押/担保/保证保险），collateral_type，guarantee_requirement，credit_history_limit（如“近2年M2≤1，无当前逾期”）；  
   - 流程 & 材料：  
     - apply_channel，required_docs，process_brief；  
   - 内部标签：  
     - fit_body_type（适合的贷款体质类型 A/B/C）、key_improvement_metric（为了匹配该产品，企业最应该提升的指标）、scenario_tag（周转/扩产/续贷/纾困等）、risk_comment。  
3. 会计实习生按产品逐个录入：  
   - 能从原始文档中直接读到的字段 → 直接填入；  
   - 读不到/不确定的字段 → 先留空，并在「待补充字段」表中记录。  

### 模块 C：沟通补齐信息（银行/中介口径）

**目标：**把文档里没有写清楚、但银行实际很看重的条件补齐。

1. 结合「待补充字段」表，为每个银行/产品准备【提纲式问题清单】，例如：  
   - 额度到底按什么算？最近12个月发票额？纳税额？流水？  
   - 今年 vs 去年的票税下滑多少会被大幅缩额度？  
   - 对连续零申报/负申报的容忍度？  
   - 是否要求一般纳税人？注册年限？  
2. 由会计实习生（必要时松阳陪同）通过电话/微信/面谈和客户经理沟通，获取口径；  
3. 将沟通结果填回《产品总表.xlsx》相应字段，并在 `risk_comment` 或备注里标注“KM 口径：xxxx”。  
4. 同步更新 `查询要求速览.xlsx`：  
   - 只保留最关键信息（便于快速查），例如：最低开票额、最低纳税额、经营年限、额度上限等。

### 模块 D：产品测试与验证记录

**目标：**验证产品是否真能用、与「体质诊断结果」的匹配度如何。

1. **纸面测试（高频，低成本）**  
   - 与松阳一起定义 3–5 个标准「企业画像」：  
     - 例如：A1 = 高开票高纳税优质体质；B1 = 正常开票但流水结构乱；C1 = 长期零申报…  
   - 会计实习生使用这些画像，对重点产品（税贷/发票贷/核心流贷）进行纸面判断，填表：  
     - 是否满足明面条件？  
     - 理论额度区间？  
     - 存在的主要风险点/不确定点？  
   - 输出一张《产品×体质矩阵表》。  
2. **真实案例测试（低频，高价值）**  
   - 当某产品被松阳认定为「主推产品」，且有真实企业愿意尝试：  
     - 由松阳/业务员负责整体沟通；  
     - 会计实习生负责过程记录：提交材料清单、审批时间、最终额度/利率、被问到的关键问题；  
   - 为每个产品建立《产品实战案例-产品名-企业代号-日期.md》，放入 `DCC2025/01_银行政策库/04_产品测试与案例/`。  

### 模块 E：与技术实习生协同

**目标：**让产品数据真正进入系统，被贷查查/AI 使用。

1. 与技术实习生一起确认：《产品总表.xlsx》每一列的字段名、类型、取值范围；  
2. 当字段定义调整时，及时同步修改：  
   - 00_数据结构说明.md  
   - 产品总表中的列名；  
3. 为技术同学提供：  
   - 小样本 JSON 导出（几条产品记录）  
   - 字段解释（例如“invoice_requirement”如何解析成可计算规则）；  
4. 对系统侧产品推荐结果做 sanity check：  
   - 当贷查查给某一企业推荐产品时，会计实习生核对是否合理；  
   - 不合理的地方，记录「产品匹配问题」，用于调整规则或字段。

## 四、分阶段执行计划（WHEN）

### 阶段 0：到岗前 1 天（由松阳准备）

- 在 GitHub 两个仓库中创建好相关目录：  
  - `DCC2025/01_银行政策库/01_原始资料/`  
  - `DCC2025/01_银行政策库/02_产品总表.xlsx`（空表，只有标题行）  
  - `DCC2025/01_银行政策库/03_查询要求速览.xlsx`（空表）  
  - `DCC2025/01_银行政策库/04_产品测试与案例/`  
- 把现有本地文件初步拷贝到 `01_原始资料/`（不要求完美命名，留给实习生整理）。

### 阶段 1：第 1 周——熟悉 & 归档

1. 阅读与理解（1–2 天）：  
   - 《资金体检与贷款体质手术-总路径图》  
   - 《资金体检医生-定位与标准话术》  
   - 《贷款体检7连问完整版v1.0》  
   - 《诊断问卷v1.0》  
2. 目录梳理（2–3 天）：  
   - 按模块 A 的要求，完成：  
     - 原始资料文件重命名；  
     - 《原始资料目录表》初版。  
3. 与技术实习生 30–60 分钟对齐一次：  
   - 共同确认《产品总表.xlsx》的字段列表；  
   - 明确哪些字段必须第一轮就录、哪些可以先空着。

### 阶段 2：第 2–3 周——产品总表首版 & 部分补齐

1. 以 `00.产品汇总.xlsx` 为主，完成首批产品录入：  
   - 优先录入：税贷、发票贷、核心流贷、常用线上小微信贷；  
   - 目标：录入 **20–30 个核心产品**。  
2. 将空白/不确定字段记录到「待补充字段」表；  
3. 开始对接银行/中介客户经理，按优先级补齐关键信息；  
4. 每周产出：  
   - 更新版《产品总表.xlsx》；  
   - 更新版《查询要求速览.xlsx》；  
   - 每周小结（本周新增产品数、已补齐字段、问题清单）。

### 阶段 3：第 4–6 周——纸面测试 + 实战记录

1. 和松阳一起确定 3–5 个标准企业画像（A/B/C 体质）。  
2. 用这些画像，对已录入的核心产品做纸面测试，输出《产品×体质矩阵表》。  
3. 对于已进入实战的企业：  
   - 建立 `产品实战案例` 文档；  
   - 记录真实批复额度、利率、流程耗时。  
4. 与技术实习生联合检查：系统侧产品推荐 vs 手工判断 是否一致，记录差异。

### 阶段 4：持续迭代（第 2 个月起）

1. 新产品录入：  
   - 每有新银行/新产品资料进来，优先判断是否值得录入；  
   - 值得录 → 按同样流程录入 + 测试；  
   - 不值得录 → 保留原始资料，标记为「观察中」。  
2. 定期清理 & 归档：  
   - 每月一次，清理废弃/重复的产品；  
   - 更新产品状态（在售/停售/政策收紧）。  
3. 支持内容输出：  
   - 为松阳的对外内容（文章/视频）提供真实产品案例与数字。

## 五、每日 / 每周节奏（HOW）

### 5.1 每日节奏（建议）

- 上午：  
  - 文档整理、产品录入、数据清洗；  
- 下午：  
  - 电话/微信向银行/中介确认条件；  
  - 完成纸面测试或实战记录；  
- 收尾：  
  - 更新当天进度到一个简单的「工作日志」：  
    - 新增产品条数 / 更新字段数 / 沟通了哪些银行。

### 5.2 每周节奏

- 周一：与松阳对齐本周重点产品 / 银行；  
- 周中：推进录入 + 沟通 + 测试；  
- 周五：  
  - 输出 1 份《本周产品库进展小结》；  
  - 与松阳 & 技术实习生 30 分钟例会，讨论：  
    - 发现的问题；  
    - 字段/规则是否需要调整；  
    - 下周优先级。

## 六、质检标准（DONE 的定义）

1. 任意一个产品，只要在《产品总表》里出现，就必须：  
   - 至少有：银行 / 产品名 / 产品类型 / 区域 / 客户类型 / 基本额度区间 / 关键票税要求；  
   - 有对应的「原始资料路径」。  
2. 对重点产品（松阳指定）：  
   - 必须完成一次纸面测试；  
   - 有明确的“适合哪些体质 A/B/C”的判断。  
3. 每周进展可量化：  
   - 新增或更新产品条目数；  
   - 补齐字段数量；  
   - 沟通过的银行/中介数量。  

> 只要按这个节奏坚持几周，  
> 银行产品这摊“散文件 + 中介口径”，  
> 就会逐渐被整理成贷查查的「产品大脑」。
