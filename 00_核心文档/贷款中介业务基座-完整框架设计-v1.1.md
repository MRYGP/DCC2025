## 第一部分:核心定位(为什么这样设计)

### 1.0 核心流程 (30秒理解基座)

**最简洁的描述**:

```
输入: 客户资料 (13个字段)
   ↓
处理: 系统诊断 + 匹配
   ↓
输出 (三段式):
├─ 路径A: 立刻进件 (匹配度>80%)
├─ 路径B: 改造后进件 (匹配度40-80%)
│   ├─ 3个月改造方案 + 效果预览
│   └─ 6个月改造方案 + 效果预览
└─ 路径C: 暂不建议 (匹配度<40%) + 复活条件
   ↓
执行: 任务追踪 + 【强制证据】管理
   ↓
复盘: 定期复评 + 失败/成功沉淀
   ↓
循环: 回到"处理"(持续陪跑)
```

**关键理念**: 
- ❌ 不是"一次性诊断报告生成器"
- ✅ 而是"持续陪跑的改造执行系统"
- 核心区别: 诊断 → 方案 → **任务** → **【强制证据】** → **复评** → 再诊断(闭环)

**v1.1新增**:
- ✅ **三段式输出**：不只告诉"能贷多少",还要告诉"不建议做什么"
- ✅ **强制证据纪律**：任务完成=必须上传证据,无证据=不能标记完成
- ✅ **合规贯穿全层**：授权、最小采集、审计留痕、可导出

### 1.1 产品定位

**❌ 我们不是什么?**
- 不是贷款平台(不和融360竞争)
- 不是简单的"效率工具"(不只是提升交付速度)
- 不是"新客户获取"工具(不只是做流量导入)

**✅ 我们是什么?**
- **存量客户复活引擎** + **改造执行系统**
- 帮中介管理"前女友"(历史客户)的系统
- 把"以前没成交"变成"可激活的资产"
- 把"改造建议"变成"可执行的任务"
- 把"时间流逝"变成"体质改善"

**核心隐喻："前女友管理系统"**

中介手里有大量"以前没成交"的客户:
- 当时资质不够(成立时间短、开票不够、征信不好等)
- 但关系还在、需求明确、知根知底
- 时间过去了,可能条件改善了
- 但中介记不住、没系统管理

这些客户就像"前女友":
- ✅ 熟人,信任基础好
- ✅ 基本情况了解(不需要重新调研)
- ✅ 付费意愿明确(想贷但当时贷不到)
- ✅ 沟通成本低(有联系方式,知道痛点)

相比之下,"新客户"其实是"别人的前女友":
- ❌ 情况不明(可能被其他中介接触过)
- ❌ 需要大量试探(凭什么比别人强?)
- ❌ 转化率低(不知道经历了什么)
- ❌ 获客成本高(广告/地推/时间)

**结论**: 与其花大力气追"新欢",不如先把"前女友"盘活。

### 1.2 核心突破点

**中介真正的焦虑**

❌ 不是: 进件不标准、返工多(效率问题)
- → "单子多,加班就行"

✅ 而是: 没有新单子(获客问题)
- → "没单子,才是大问题"

**存量客户的商业价值**

对比分析:

| 维度 | 存量客户(前女友) | 新客户(可能是别人的前女友) |
|-----|---------------|----------------------|
| 信任基础 | ✅ 有(聊过、见过) | ❌ 无(需要建立) |
| 情况了解 | ✅ 知根知底 | ❌ 情况不明 |
| 付费意愿 | ✅ 明确(想贷但条件不够) | ⚠️ 不确定 |
| 沟通成本 | ✅ 低(一个电话) | ❌ 高(需要试探) |
| 转化率 | ✅ 高(20-30%) | ❌ 低(5-10%) |
| 获客成本 | ✅ 0元(已有联系方式) | ❌ 高(广告/地推) |
| ROI | ✅ 清晰可量化 | ❌ 不确定 |

**ROI计算示例**

假设中介有200个历史客户:
- A类(现在能贷): 40个 × 20%成交率 = 8个 × 3万佣金 = 24万
- B类(改造后能贷): 60个 × 30%愿意改造 = 18个
  - 改造服务费: 18个 × 7万(平均) = 126万(我们分润70% = 88万,中介分30% = 38万)
  - 改造后成交: 18个 × 80%成交率 = 14个 × 4万佣金 = 56万

**中介年收入**:
- A类佣金: 24万
- B类改造分润: 38万
- B类佣金: 56万
- 合计: 118万
- 投入: 工具费3.6万/年
- **ROI: 118万 / 3.6万 = 32.8倍**

**我们的年收入(单个中介)**:
- 工具订阅: 3.6万
- 改造服务: 88万
- **合计: 91.6万/年/中介**

关键洞察: **改造服务收入 >> 工具订阅收入**

### 1.3 设计原则

**原则1: 数据驱动**
- 所有功能基于数据流
- 数据可追溯、可导出、可复盘
- 数据积累越多,系统越准

**原则2: 闭环设计**
- 不是"一次性诊断",而是"持续陪跑"
- 诊断 → 改造 → 复评 → 再诊断(循环)
- 失败 → 沉淀 → 优化 → 再尝试(循环)
- 成功 → 复制 → 扩展 → 更多成功(循环)

**原则3: 可扩展**
- 分层架构,每层独立可优化
- 模块化设计,模块间松耦合
- 配置化驱动,不需要改代码
- 从V0简单规则 → V1复杂规则 → V2机器学习

**原则4: 可追溯**
- 所有操作留痕(谁、什么时候、做了什么)
- 时间线可视化(所有动作一目了然)
- 失败原因结构化(知道为什么不行)
- 成功案例可复制(知道为什么行)

**原则5: 可执行性 (避免成为报告生成器)**

这是基座最关键的原则！

- ❌ 不是: 生成漂亮的诊断报告就结束
- ✅ 而是: 报告 → 任务 → 证据 → 追踪 → 复盘(完整闭环)

**关键设计点**:
1. **改造方案必须拆解成任务**
   - 不是"建议开票提升到50万"
   - 而是"任务1: 联系服务商(责任人: 中介, 截止: X月X日)"
   
2. **任务必须有证据要求** ⭐ v1.1强化
   - 不是"完成了"
   - 而是"上传发票复印件(版本1, 上传时间: X月X日)"
   - **强制规则**: 无证据=不能标记完成
   
3. **必须有复盘机制**
   - 不是"3个月后看看有没有效果"
   - 而是"系统自动触发复评, 生成变化报告, 记录原因"

**GPT的警告**: 如果没有"执行追踪"和"复盘机制"，基座会变成"漂亮的报告生成器"，很难真的陪跑起来。

**原则6: 合规与可信 (v1.1新增)** ⭐

这是系统长期可持续的基础！

- ❌ 不是: 无限制采集数据
- ✅ 而是: 最小必要采集 + 授权 + 可导出 + 留痕

**关键设计点**:
1. **最小化采集**: 只采集13个必需字段,不过度收集
2. **授权与说明**: 明确告知数据用途,获得授权
3. **审计留痕**: 所有操作记录(谁/何时/改了什么)
4. **可导出/删除**: 中介可随时导出或要求删除数据

**PIPL合规要求**: 个人信息处理必须遵循合法、正当、最少必要、目的明确、透明度等原则。

---

**关键区别: 基座 vs Demo**

| 维度 | Demo(演示原型) | 基座(完整系统) |
|-----|-------------|-------------|
| 目标 | 展示概念 | 真实可用 |
| 数据 | 假数据、硬编码 | 真实数据、数据库 |
| 流程 | 单向流程 | 完整闭环 |
| 追溯 | 无历史记录 | 所有操作留痕 |
| 扩展 | 改代码 | 改配置 |
| 迭代 | 推倒重来 | 渐进演进 |
| 合规 | 无要求 | 符合PIPL |

#### 模块设计

**模块1: 最小化采集**
- 字段定义: 13个字段都是"业务必需",无冗余采集
- 数据来源标注: 记录数据从哪里来(手工输入/Excel/OCR/API)
- 用途说明: 每个字段旁边显示"为什么要填这个"

**模块2: 数据授权**
- 授权勾选: 客户资料录入时,勾选"授权中介处理信息"
- 授权内容: 明确说明数据用途(诊断、匹配、改造、复盘)
- 授权期限: 授权有效期(默认1年)

**模块3: 审计留痕**
- 操作日志: 所有操作记录(谁/何时/做了什么)
- 数据访问日志: 谁查看了客户档案
- 数据修改日志: 谁修改了哪些字段(old_value → new_value)
- 保留期限: 日志保留3年

**模块4: 可解释性**
- 字段说明: 每个字段用途说明
- 算法说明: 健康分/匹配结果如何计算
- 决策透明: 为什么推荐这个方案

**模块5: 可导出/删除**
- 数据导出: Excel/PDF/JSON格式
- 删除请求: 客户要求删除时,标记为"已删除",保留日志
- 数据迁移: 支持中介带走数据

CREATE TABLE data_authorizations (

);

- "我的数据会不会被泄露?"
- "我能不能带走自己的数据?"

**合规要求**
- PIPL(个人信息保护法)要求
- 最小化采集、目的明确、透明度
- 数据安全、可导出权

**长期可持续**
- 现在单租户模式,合规压力小
- 未来SaaS化,合规是必备能力
- 提前设计,避免推倒重来

#### 3.3.1 模块设计

**模块1: 健康分计算**
- 输入: 13个字段
- 算法: 加权求和(V0简单规则)
- 输出: 
  - 总分(0-1000分)
  - 等级(A/B/C/D)
  - 6个维度得分(流水、票税、征信、资产、资质、行业)
  - 维度雷达图

**模块2: 三段式输出 (v1.1核心更新)** ⭐

**设计理念**:
- ❌ 不只告诉"能贷多少"
- ✅ 还要告诉"不建议做什么",避免浪费时间

**三段式结构**:

```
路径A: 立刻进件 (匹配度 >80%)
├─ 符合产品列表 (2-5个)
├─ 每个产品的额度/利率/期限
├─ 必需材料清单
└─ 预计审批时间

路径B: 改造后进件 (匹配度 40-80%)
├─ 3个月改造方案 + 效果预览
├─ 6个月改造方案 + 效果预览
├─ 关键证据清单
└─ 改造后预期匹配度

路径C: 暂不建议 (匹配度 <40%) ⭐ v1.1新增
├─ 核心缺陷 (征信/经营年限/负债率等)
├─ 为什么不建议 (避免踩雷)
├─ 复活触发条件 ("成立满2年后复评")
└─ 自动提醒时间 (90天后系统提醒)
```

**匹配度计算**:
```python
def calculate_match_rate(customer, product):
    """
    计算客户与产品的匹配度
    返回: 0-100分
    """
    score = 0
    total_rules = len(product.rules)
    
    for rule in product.rules:
        if evaluate_rule(customer, rule):
            score += 1
    
    match_rate = (score / total_rules) * 100
    return match_rate
```

**路径判断逻辑**:
```python
def classify_customer_path(matched_products):
    """
    根据匹配产品数量和匹配度,判断客户属于哪个路径
    """
    if len(matched_products) == 0:
        return "路径C: 暂不建议"
    
    # 计算平均匹配度
    avg_match_rate = sum(p['match_rate'] for p in matched_products) / len(matched_products)
    
    if avg_match_rate >= 80:
        return "路径A: 立刻进件"
    elif avg_match_rate >= 40:
        return "路径B: 改造后进件"
    else:
        return "路径C: 暂不建议"
```

**三段式的价值**

**1. 避免中介浪费时间**

- ✅ 我们的模式: 明确告诉"这个客户不要碰",系统会在条件成熟时提醒

**2. 给客户clear的预期**
- ❌ 传统模式: "我帮你试试"(含糊其辞)
- ✅ 我们的模式: "您现在条件不够,建议90天后再申请,否则会留查询记录"

**3. 数据复利**
- 路径C的客户不是"永远放弃"
- 而是"设置复活条件",系统自动提醒
- 把"时间流逝"变成"资产复活"

**可量化 vs 模糊评价**
- ❌ 不是"感觉不太行"
- ✅ 而是"匹配度38%,低于40%阈值,不建议申请"

**可解释性**
- 每个路径判断都能说清楚为什么
- "您符合路径A,因为匹配度85%,超过80%阈值"
- "您不符合路径C,因为虽然开票不够,但改造后匹配度可达75%"

**可行动性**
- 路径A → 立即联系客户,准备材料
- 路径B → 推荐改造方案,评估ROI
- 路径C → 设置提醒,等待时机

class Task:
    """
    任务模型 (v1.1更新)
    """
    def __init__(self, task_data):
        self.id = task_data['id']
        self.task_name = task_data['task_name']
        self.status = task_data['status']
        self.evidence_required = task_data['evidence_required']  # ⭐ 新增
        self.evidence_type = task_data['evidence_type']          # ⭐ 新增
        self.evidence_list = []  # 已上传的证据列表
    
    def mark_completed(self):
        """
        标记任务完成 (v1.1新增强制验证)
        """
        # 强制证据验证
        if self.evidence_required and len(self.evidence_list) == 0:
            raise ValidationError(
                "❌ 无法标记完成: 此任务必须上传证据\n"
                f"需要上传: {self.evidence_type}\n"
                "请先上传证据,再标记完成。"
            )
        
        # 验证通过,标记完成
        self.status = TaskStatus.COMPLETED
        self.completed_at = datetime.now()
        
        # 记录到时间线
        timeline_record = {
            'action_type': 'complete',
            'action_detail': f'完成任务: {self.task_name}',
            'metadata': {
                'evidence_count': len(self.evidence_list),
                'evidence_files': [e.file_name for e in self.evidence_list]
            }
        }
        save_to_timeline(timeline_record)
        
        return True

def upload_evidence(task_id, file):
    """
    上传证据 (v1.1新增)
    """
    task = get_task_by_id(task_id)
    
    # 验证文件格式
    allowed_formats = ['pdf', 'jpg', 'jpeg', 'png']
    file_ext = file.filename.split('.')[-1].lower()
    if file_ext not in allowed_formats:
        raise ValidationError(f"不支持的文件格式: {file_ext}")
    
    # 保存文件
    file_path = save_file(file)
    
    # 创建证据记录
    evidence = Evidence(
        customer_id=task.customer_id,
        task_id=task.id,
        evidence_type=task.evidence_type,
        file_name=file.filename,
        file_path=file_path,
        uploaded_by=current_user.name
    )
    
    # 添加到任务的证据列表
    task.evidence_list.append(evidence)
    
    # 记录到时间线
    timeline_record = {
        'action_type': 'upload',
        'action_detail': f'上传证据: {file.filename}',
        'metadata': {
            'task_name': task.task_name,
            'file_size': file.size
        }
    }
    save_to_timeline(timeline_record)
    
    return evidence

**4. 系统学习**
- 有证据 → 知道真实执行情况
- 无证据 → 不知道是"真完成"还是"假完成"
- 数据质量 → 直接影响系统优化

**可追溯性**
- 所有操作都有记录
- 谁做了什么一清二楚
- 失败了能回溯整个过程找原因

**可协作性**
- 中介和客户都知道谁负责什么
- 任务清晰分配,不会遗漏
- 进度透明,随时查看

**可复盘性**
- 时间线是完整的故事
- 成功案例可以复制流程
- 失败案例可以分析原因
