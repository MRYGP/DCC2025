## 第一部分：核心突破点（为什么做这个）

### 1.1 中介最真实的困境：没单子，不是效率低

**错误假设**：
- 中介最痛的是"进件效率低""流程不标准"
- 解决方案：给中介做一个"像融360的系统"

**真实情况**：
- 中介最痛的是"没有新单子"（获客焦虑）
- 但获取新客户成本高、转化率低、情况不明
- 很多新客户其实是"别人的前女友"（已经被其他中介接触过）

**关键洞察**：
> 与其花大力气追"新欢"（情况不明），
> 不如先把"前女友"盘活（知根知底）。

**数据对比**：

| 维度 | 存量客户（前女友） | 新客户（可能是别人的前女友） |
|------|-------------------|---------------------------|
| 信任基础 | ✅ 有（聊过、见过） | ❌ 无（需要建立） |
| 情况了解 | ✅ 知根知底 | ❌ 情况不明 |
| 付费意愿 | ✅ 明确（想贷但当时条件不够） | ⚠️ 不确定 |
| 沟通成本 | ✅ 低（一个电话） | ❌ 高（需要试探） |
| 转化率 | ✅ 20-30% | ❌ 5-10% |
| 获客成本 | ✅ 0元（已有联系方式） | ❌ 高（广告/地推） |
| ROI | ✅ 清晰可量化 | ❌ 不确定 |

### 1.2 存量客户：被忽视的金矿

**中介手里都有什么？**
- 100-200个"以前没成交"的历史客户
- 这些客户特点：
  - 知根知底（已经沟通过）
  - 有联系方式
  - 有明确的贷款需求
  - 只是当时条件不够（成立时间、开票额、征信等）

**关键问题**：
- 时间过去了，可能条件够了
- 但中介：
  - 记不住这么多客户
  - 不知道谁现在可以贷了
  - 不知道谁需要等多久
  - 没有系统提醒

**结果**：
- 这些"前女友"睡在Excel里
- 偶尔翻翻，靠运气
- 大部分就这样流失了

**场景还原**：
```
2023年1月：A公司找中介贷款
当时：成立1.5年，不符合"成立≥2年"的要求
中介：没办法，只能等

2024年7月：A公司成立满2年了
问题：中介记得吗？会主动联系吗？

答案：大概率不会
```

### 1.3 我们的价值主张

**核心定位**：
> 存量客户复活引擎 + 改造执行系统

**不是什么**：
- ❌ 不是贷款平台（不和融360竞争流量）
- ❌ 不是传统助贷CRM（线索管理+进件管理）

**是什么**：
- ✅ 帮中介把200个"前女友"变成可管理的资产
- ✅ 把"改造建议"变成"可执行的任务+证据"
- ✅ 把"时间流逝"变成"体质改善+自动提醒"

**与竞品差异**：
- 传统助贷CRM：线索管理 + 进件管理 + 跟单管理（聚焦新客户获取和效率提升）
- 我们：**存量客户复活 + 改造执行闭环**（市场上相对少见）

**核心价值**：
- 对中介：把睡在Excel里的200个客户，激活20%，就是40个客户
- 对客户：不是"你不够格"，而是"3个月后你就够格了"
- 对我们：积累失败→成功的改造数据，形成护城河

### 1.4 商业逻辑：双重变现

**场景1：条件够了（40%）**
```
系统诊断 → A类客户（现在能贷）
中介跟进 → "王总，现在条件够了，要不要试试？"
成交 → 中介赚佣金（3万/单）
```

**场景2：条件不够，但愿意改造（30%）**
```
系统诊断 → B类客户（缺口：开票不够）
改造方案 → "李总，我们有科学的改造方案，
            3个月内把开票提升到50万，
            然后能贷到150万，要不要试试？"
改造服务 → 我们提供H业务，收8万
3个月后成交 → 中介再赚佣金（4万/单）

总收入：改造服务费（5-8万）+ 贷款佣金（4万）= 9-12万
```

**场景3：条件不够，暂不改造（30%）**
```
系统诊断 → B类客户（缺口：成立时间不够）
改造方案 → "张总，再等2个月就满2年了，
            到时候能申请到80万。
            我给您设个提醒，2个月后我再联系您。"
2个月后自动提醒 → 中介跟进 → 成交
```

**收入模型（单个中介）**：
```
假设200个存量客户：
- A类（40个）→ 成交8个 → 佣金24万
- B类（60个）→ 改造18个 → 改造费126万（我们88万+中介38万）
                         → 改造后成交14个 → 佣金56万
- C类（100个）→ 暂不跟进

中介总收入：24 + 38（改造分润） + 56 = 118万
我们总收入：3.6万（工具费）+ 88万（改造服务）= 91.6万

关键：改造服务收入 >> 工具订阅收入
```

## 第二部分：六层架构（系统设计核心）

### 2.0 第0层：合规与可信（贯穿能力）

这不是一个独立的功能层，而是**系统级能力**，贯穿所有层。

**为什么需要第0层？**
- 系统会处理企业主和法人信息
- 中介最担心：数据泄露、被抢客户、迁移困难
- 合规要求：PIPL（个人信息保护法）

**核心原则**：

1. **最小化采集**
   - 只收集13个必要字段
   - 每个字段都说清楚用来干什么
   - 不收集与贷款无关的信息

2. **授权留档**
   - 客户授权书
   - 用途说明
   - 处理记录

3. **操作留痕**
   - 谁在什么时候做了什么
   - 所有操作可追溯
   - 审计日志

4. **数据可导出**
   - 中介可以随时导出所有数据
   - 支持Excel/PDF/JSON格式
   - 切换成本低（不锁定客户）

5. **可解释性**
   - 每个分数都能说清楚怎么来的
   - 每个缺口都能说清楚差多少
   - 每个方案都能说清楚为什么推荐

6. **合规底线**
   - ⚠️ 改造方案红线：严禁虚假材料、虚构交易、循环开票等
   - ✅ 合规动作：真实业务改善、合法证据标准、可持续经营

**V0简化版**（P1）：
- ✅ V0必须有：操作留痕（基础时间线）+ 数据可导出（Excel/PDF）
- ⏸️ V1补全：授权管理 + 审计日志完整版

## 第三部分：三大闭环（系统如何运转）

### 3.1 闭环1：诊断-改造-复评（核心业务闭环）

这是**最核心的业务闭环**，把"一次性诊断"变成"持续陪跑"。

```
Step 1：批量导入历史客户（200个）
↓
Step 2：系统自动诊断分层
• A类（40个）→ 现在能贷
• B类（60个）→ 需要改造
• C类（100个）→ 暂不跟进
↓
Step 3：分类处理
• A类 → 立即提醒，推荐产品
• B类 → 生成改造方案 + 效果预览
• C类 → 设置复活触发条件
↓
Step 4：执行追踪（B类客户）
• 拆解任务（动作+证据+负责人）
• 上传证据
• 记录时间线
↓
Step 5：定期复评（3个月后）
• 重新计算分数
• 查看变化（分数/产品/额度）
• 更新分类（B类→A类）
↓
Step 6：回到Step 2（循环）
```

**关键点**：
- 不是一次性诊断，而是持续陪跑
- 每次复评都是新一轮诊断的开始
- 系统会自动提醒"该复评了"

**场景示例**：
```
2025-01-26：批量导入200个历史客户
• A类40个 → 立即跟进 → 成交8个
• B类60个 → 推荐改造方案
• C类100个 → 设置提醒

2025-04-26：3个月后
• B类复评 → 18个完成改造
• 健康分提升 → 变成A类
• 成交14个

2025-07-26：6个月后
• C类复评 → 部分条件满足
• 新增A类客户10个
```
