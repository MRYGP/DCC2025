~~~markdown
### 1.2 存量客户：被忽视的金矿

[写这一节时要说清楚]

**中介手里都有什么？**
- 100-200个"以前没成交"的历史客户
- 这些客户特点：
  - 知根知底（已经沟通过）
  - 有联系方式
  - 有明确的贷款需求
  - 只是当时条件不够（成立时间、开票额、征信等）

**关键问题**：
- 时间过去了，可能条件够了
- 但中介：
  - 记不住这么多客户
  - 不知道谁现在可以贷了
  - 不知道谁需要等多久
  - 没有系统提醒

**结果**：
- 这些"前女友"睡在Excel里
- 偶尔翻翻，靠运气
- 大部分就这样流失了

**场景还原**：
```
2023年1月：A公司找中介贷款
当时：成立1.5年，不符合"成立≥2年"的要求
中介：没办法，只能等

2024年7月：A公司成立满2年了
问题：中介记得吗？会主动联系吗？

答案：大概率不会
```

### 1.3 我们的价值主张

[写这一节时要明确定位]

**核心定位**：
> 存量客户复活引擎 + 改造执行系统

**不是什么**：
- ❌ 不是贷款平台（不和融360竞争流量）
- ❌ 不是传统助贷CRM（线索管理+进件管理）

**是什么**：

- ✅ 把"改造建议"变成"可执行的任务+证据"
- ✅ 把"时间流逝"变成"体质改善+自动提醒"

**与竞品差异**：
- 传统助贷CRM：线索管理 + 进件管理 + 跟单管理（聚焦新客户获取和效率提升）
- 我们：**存量客户复活 + 改造执行闭环**（市场上相对少见）

**核心价值**：
- 对中介：把睡在Excel里的200个客户，激活20%，就是40个客户
- 对客户：不是"你不够格"，而是"3个月后你就够格了"
- 对我们：积累失败→成功的改造数据，形成护城河

### 1.4 商业逻辑：双重变现

[写这一节时要说清楚商业模式]

**场景1：条件够了（40%）**
```
系统诊断 → A类客户（现在能贷）
中介跟进 → "王总，现在条件够了，要不要试试？"
成交 → 中介赚佣金（3万/单）
```

**场景2：条件不够，但愿意改造（30%）**
```
系统诊断 → B类客户（缺口：开票不够）
改造方案 → "李总，我们有科学的改造方案，
            3个月内把开票提升到50万，
            然后能贷到150万，要不要试试？"
改造服务 → 我们提供H业务，收8万
3个月后成交 → 中介再赚佣金（4万/单）

总收入：改造服务费（5-8万）+ 贷款佣金（4万）= 9-12万
```

**场景3：条件不够，暂不改造（30%）**
```
系统诊断 → B类客户（缺口：成立时间不够）
改造方案 → "张总，再等2个月就满2年了，
            到时候能申请到80万。
            我给您设个提醒，2个月后我再联系您。"
2个月后自动提醒 → 中介跟进 → 成交
```

**收入模型（单个中介）**：
```
假设200个存量客户：
- A类（40个）→ 成交8个 → 佣金24万
- B类（60个）→ 改造18个 → 改造费88万 + 佣金56万
- C类（100个）→ 暂不跟进

中介总收入：24 + 38（改造分润） + 56 = 118万
我们总收入：3.6万（工具费）+ 88万（改造服务）= 91.6万

关键：改造服务收入 >> 工具订阅收入
```

   - 中介可以随时导出所有数据

   - ⚠️ 改造方案红线：严禁虚假材料、虚构交易、循环开票等

**历史记录的价值**：
- 不只是"当下状态"，而是"变化轨迹"
- 可以分析：哪些改造真的有效
- 可以复盘：为什么失败、为什么成功

### 2.3 第2层：规则引擎层

[详细说明这一层的设计]

#### 2.3.1 核心功能

**功能1：产品库**

每个产品包含：
- 基本信息：银行名称、产品名称、类型（税贷/发票贷/抵押贷等）
- 额度信息：最低/最高额度、利率区间、期限
- 准入条件：规则化表达
  - 成立时间 ≥ X年
  - 年纳税额 ≥ X万
  - 征信要求（无M3+逾期）
  - 行业限制（不支持XX行业）
- 额度算法：年纳税额 × 倍数（或其他公式）
- 材料要求：必需/可选/加分

V0产品库：5-10个核心产品
V1产品库：20+个产品
V2产品库：支持中介自定义

**功能2：匹配引擎**

输入：客户13个字段
规则：产品准入条件逐一对比
输出：
- 符合哪些产品（按匹配度排序）
- 每个产品的匹配度分数
- 不符合的原因

伪代码：
```python
def match_products(customer):
  matched = []
  for product in product_library:
    score = 0
    missing = []
    for rule in product.rules:
      if customer满足rule:
        score += 1
      else if rule.is_required:
        missing.append(rule)
        break  # 硬性条件不符合，直接跳过
      else:
        missing.append(rule)
    
    if len(missing硬性条件) == 0:
      matched.append({
        product: product,
        score: score,
        missing: missing
      })
  
  return sort(matched, by=score)
```

**功能3：缺口识别引擎**

输入：客户数据 + 产品要求
对比：逐条对比找差距
输出：
- 缺什么（成立时间、开票额、纳税额等）
- 差多少（具体数值）
- 影响哪些产品
- 可改造性（时间型/操作型/困难型）

示例：
```
缺口1：开票不够 ⭐⭐⭐ 高影响
├─ 当前：30万/年
├─ 需要：50万/年
├─ 差距：20万
├─ 影响产品：XX银行发票贷（额度100万）
└─ 可改造性：操作型（3-6个月）

缺口2：成立时间不够 ⭐⭐ 中影响
├─ 当前：1.8年
├─ 需要：2年
├─ 差距：2.4个月
├─ 影响产品：XX银行流贷（额度80万）
└─ 可改造性：时间型（等待）
```

### 2.4 第3层：诊断输出层

[详细说明这一层的设计]

#### 2.4.1 核心功能

**功能1：贷款能力评分（替代"健康分"）**

为什么不叫"健康分"？
- 容易被误解为玄学总分
- 不够具体、可解释

贷款能力评分：
- 0-1000分
- 基于13个字段的规则计算
- 可细分：
  - 进件质量分（材料完整度、证据有效性）
  - 资质体质分（开票、纳税、流水、征信等）

算法（V0简单版）：
```
基础分：200分（企业存在）

维度1：流水体质（200分）
- 对公流水 >= 100万/年 → 200分
- 50-100万 → 150分
- < 50万 → 100分

维度2：票税体质（200分）
- 年开票额 >= 500万 → 100分
- 年纳税额 >= 10万 → 100分

维度3：征信体质（200分）
- 法人征信好 → 100分
- 企业征信好 → 100分

维度4：资产体质（100分）
- 有抵押物 → 100分

维度5：资质体质（100分）
- 成立时间 >= 3年 → 50分
- 成立时间 >= 2年 → 30分
- （其他资质认定在V1加入）

   - 多头授信（同时申请多家）
   - 逾期记录（M1/M3/M6）
   - 查询次数过多
   - 小贷占比过高

2. 财务类
   - 流水不够
   - 开票不够
   - 纳税不够
   - 流水不稳定
   - 收入成本不匹配

3. 资质类
   - 成立时间不够
   - 无高新认定
   - 无专精特新认定
   - 行业不符合

4. 资产类
   - 无抵押物
   - 抵押物价值不足
   - 无担保人

5. 其他
   - 资料不全
   - 客户配合度低
   - 银行政策变化

每次失败必须记录：
- 失败产品
- 失败原因（多选）
- 失败时间
- 备注（银行具体反馈）

统计分析：
- 最常见失败原因
- 不同产品的失败率
- 不同客户类型的失败率

**功能3：成功案例库**

记录内容：
- 客户画像（行业、规模、资质）
- 改造方案（用了哪个方案）
- 执行周期（实际花了多久）
- 最终结果（成交金额/利率）
- 关键成功因素（哪些步骤最关键）

作用：
- 话术参考（给中介看成功案例）
- 方案优化（哪些方案成功率高）
- 经验复制（类似客户可以参考）

**功能4：数据导出**

导出内容：
- 客户档案（所有字段）
- 时间线记录（所有操作）
- 任务清单（已完成/未完成）
- 材料清单（所有文件）
- 评估报告（PDF格式）

导出格式：
- Excel（客户档案、任务清单）
- PDF（评估报告、时间线）
- JSON（完整数据，可导入其他系统）

作用：
- 数据安全（中介可以随时带走数据）
- 合规要求（客户有权要求导出数据）
- 系统迁移（降低切换成本）

## 第三部分：三大闭环（系统如何运转）

### 3.1 闭环1：诊断-改造-复评

[详细说明这个闭环]

这是**最核心的业务闭环**，把"一次性诊断"变成"持续陪跑"：
```
Step 1：输入客户资料
↓
Step 2：系统诊断
- 计算健康分
- 三段式输出：现在能做/值得改造/不建议做
↓
Step 3：生成改造方案
- 3个月方案 + 效果预览
- 6个月方案 + 效果预览
- 方案对比 + ROI计算
↓
Step 4：执行追踪
- 拆解任务（动作+证据+负责人）
- 上传证据
- 记录时间线
↓
Step 5：定期复评
- 重新计算健康分
- 查看变化（分数/产品/额度）
- 更新方案
↓
Step 6：回到Step 2（循环）
```

**关键点**：
- 这不是一次性诊断，而是持续陪跑
- 每次复评都是新一轮诊断的开始
- 系统会自动提醒"该复评了"
- 改造是可量化的：+100分 = +100万额度

**场景示例**：
```
2025-01-25：
- 诊断：健康分650，当下能贷130万
- 方案：开票提升术（3个月）
- 任务：收集材料、联系H业务团队

2025-04-25：
- 复评：健康分750，新增2个产品
- 结果：总可贷额度310万（+180万）
- 成交：客户申请发票贷100万

2025-07-25：
- 复评：继续高新认定
- 方案：再等3个月
- 提醒：2025-10-25再联系
```
~~~
